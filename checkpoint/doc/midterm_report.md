## 一、课题进展情况

目前本课题已经完成了从问题定义、方案设计到部分系统实现和实验验证的前半段工作，整体进度已经从“选题调研”推进到“系统跑通并得到初步结果”的阶段。前期首先围绕论文研究问题进行了集中调研和梳理，主要包括以下内容：一是调研了大语言模型在线推理的基本执行过程，重点整理了 Prefill 与 Decode 两阶段在计算特征、资源占用和时延表现上的差异；二是调研了 PD 共置部署下混合批处理、Chunked Prefill、KV Cache 竞争等机制，分析为什么在该场景中仅靠队列长度、运行请求数、总 Token 数等静态指标难以准确反映实例未来服务能力；三是调研了集群级全局调度的典型目标与评价指标，明确以 TTFT 和 TPOT 作为核心服务质量指标；四是调研了现有离线性能模拟器与在线调度方法，重点比较了基于实例负载混合指标的方法与基于请求性能预测的方法，并整理了两类方法各自的优缺点和适用条件。

在完成上述调研后，已经基本明确本文的研究主线，即不再继续围绕少量实例负载指标做人工组合和调参，而是将原本用于离线分析的性能模拟能力改造为在线可调用的性能预测模块，在请求到达时直接预测该请求在候选实例上的执行表现，并用预测得到的 TTFT 结果指导全局调度。围绕这一主线，论文的绪论和背景章节已经完成主体撰写，当前已经形成较完整的问题提出逻辑：先说明 MaaS 场景下大模型服务面临的高并发低时延压力，再说明 PD 共置场景中实例负载与请求时延之间关系复杂，最后引出“将离线模拟器在线化”这一论文切入点。到目前为止，论文前两章已经不再是提纲状态，而是形成了较为完整的正文内容，能够支撑中期答辩中对研究背景、研究意义、研究问题来源和现有工作不足的说明。

除文字撰写外，前期还完成了基于已有实验和图像材料的初步问题论证。例如，已经整理出 Company-X 类混合指标调度策略在不同参数取值下的性能变化图，并在论文中明确写出：在 ChatBot 工作负载上，当参数偏离最优值时，TPOT Mean 增加 30.1%，TPOT P95 增加 32.3%。这部分结果已经较清楚地说明，传统混合指标方法不仅表达能力有限，而且高度依赖参数调节，作为本文提出预测驱动调度方案的动机支撑已经比较充分。

在系统设计方面，目前已经完成在线性能预测模块的总体架构设计，并将其细化为可以直接落地的接口与状态管理机制。具体而言，已经完成了 `Predict`、`Add_request` 和 `Sync_State` 三个核心接口的定义及作用划分：`Predict` 负责在请求到达时对每个候选实例进行反事实预测；`Add_request` 负责在调度结果确定后更新预测模块中的实例演进状态；`Sync_State` 负责在推理实例产生真实执行结果后，将真实状态同步回调度器侧。围绕这三个接口，已经完成了在线场景下的状态管理设计，提出并写入论文的是真实状态、演进状态和模拟状态三层机制。其中，真实状态用于与实例保持一致，演进状态用于复用前序请求的预测推进结果，模拟状态用于承载当前请求的临时预测过程。这个设计已经不只是概念描述，而是明确对应到后续实现逻辑，能够解释为什么在线预测既能避免污染真实状态，又能减少重复模拟开销。

在系统实现方面，已经完成全局调度器、状态同步通路和性能预测器主体的工程落地。当前全局调度器使用 Rust 实现，并基于 Tokio 异步运行时支持多实例并发预测；为了便于公平比较，也已经将多种基线策略统一接入同一调度框架，包括 vLLM、Dynamo、Company-X 和本文实现的 Predictor。与此同时，已经对 vLLM 推理框架进行了低侵入式改造，修改位置主要集中在 Apiserver 和 Scheduler 两个组件：在 Apiserver 侧新增 SSE 长连接接口，用于持续上报实例状态；在 Scheduler 侧对批处理构建过程和状态更新过程增加埋点，能够记录每轮批处理中的请求组成、处理长度、生成 Token 情况、请求完成情况以及 KV Cache 的分配与剔除变化。也就是说，目前并不是只有“论文设计”，而是系统的数据通路已经打通，调度器可以接收实例增量状态并据此更新自身维护的状态镜像。

在性能预测器部分，已经按照“离线剖析 + 在线查表”的思路完成主体实现。对不同类型模块已经明确了特征选取方式，例如对矩阵计算类模块主要使用 Token 数作为特征，对 Attention 部分则分别对 Prefill 和 Decode 核函数建模，并使用 KV Cache 大小、计算量、请求数和平均 KV Cache 大小等特征进行映射。当前论文中已经明确写出：在线预测阶段主要执行若干次查表操作，预测平均开销被控制在微秒级，并在总结中给出“平均时延可控制在 50 微秒以下”的阶段性结论。这说明预测模块已经具备进入调度关键路径的工程可行性，而不是停留在理论讨论层面。

在实验评估方面，已经完成端到端实验平台搭建、基线实现和一批代表性结果采集。当前实验环境为 8 张 NVIDIA H20 GPU 组成的推理集群，全局调度器、请求重放器和模型实例部署在同一台具有 176 个 Intel Xeon CPU 核心和 1TB DRAM 的服务器上；推理引擎使用 vLLM-v1，Attention 后端使用 FlashInfer。实验模型已经覆盖 Dense 架构的 Qwen2.5-7B 和 MoE 架构的 Qwen3-30B-A3B，工作负载覆盖 ChatBot、Coder、Agent 和 ToolAgent 四类典型场景。当前已完成的端到端结果表明：在 Qwen ChatBot 负载、22 QPS 下，相比 Company-X 策略，Predictor 可带来 31.0% 的 TTFT Mean 改善和 36.3% 的 TTFT P95 改善；在 Qwen Coder 负载、7.5 QPS 下，相比 vLLM 策略，TTFT Mean 改善 24.0%，TTFT P95 改善 10.7%；同时，在 Qwen Coder 负载、7.5 QPS 下，相比 TPOT 表现最优或接近最优的 Company-X，Predictor 仅带来 4.4% 的 TPOT Mean 下降和 3.1% 的 TPOT P95 下降。也就是说，目前已经拿到了能够支撑论文主要论点的核心结果：预测驱动调度能够显著改善 TTFT，并且没有以严重恶化 TPOT 为代价。

从论文写作进度来看，目前《绪论》《背景知识与国内外研究现状》《系统设计》《系统实现》四章已经完成度较高，《实验评估》章节已写出实验设置和主要端到端结果分析，《总结》章节也已经形成初稿。当前剩余工作主要集中在补全实验评估中的若干子节、填补个别图和 TODO 内容、以及对全文进行系统润色和收束。

## 三、存在的问题及解决思路

1. 存在的问题：`evaluation.tex` 中预测准确性分析部分仍有多个 TODO，尚未填入具体工作负载、请求到达速率、误差分布结果和对应图像，因此目前只能证明端到端调度有效，但对“为什么有效、误差有多大”的支撑还不够完整。  
   解决思路：下一步需要从一次完整请求重放实验中导出每个请求在实际指派实例上的 `TTFT(Pred)` 与 `TTFT(Real)`，绘制误差 CDF，并至少给出 Dense 模型和 MoE 模型各一组结果。完成后要在论文中明确写出如“多少比例请求的预测误差落在多少毫秒内”这一类可量化结论。这样可以把“系统有效”进一步推进为“预测模块本身也具备可解释的准确性”。

2. 存在的问题：跨表迁移分析和 Spearman 秩相关分析目前已经有章节结构，但关键实验数据尚未补齐，包括跨表预测带来的 TTFT/TPOT 退化比例、Top-1 选择一致率和排序相关性统计结果都还没有最终落文。  
   解决思路：需要补做一组“部署 Qwen3-30B-A3B、预测时使用 Qwen2.5-7B 预测表”的实验，并同步记录正常预测与跨表预测对每个请求的候选实例排序结果。后续要从三个角度补齐分析：一是端到端性能是否仍优于 Company-X；二是两张预测表的 Top-1 选择一致率有多高；三是 Spearman 秩相关的均值和分位数如何。完成这部分后，论文将不仅能说明方法有效，还能说明该方法具有一定迁移性和工程推广价值。

3. 存在的问题：可扩展性分析部分虽然已经提出了“逻辑实例映射物理实例”的实验思路，但还没有把不同实例规模下的预测模块吞吐上界与系统服务能力上界做成完整结果，因此目前对“大规模集群下预测模块不会成为瓶颈”的论证还停留在阶段性判断。  
   解决思路：需要补充压力测试，统计预测模块在不同逻辑实例规模下的请求处理吞吐，并与 8 实例集群的服务能力上界做线性对照。最终要在论文中明确写出“在多少实例规模内预测模块吞吐仍高于系统服务能力上界”，这样老师在看报告时能清楚知道这部分工作不是停留在设想，而是有明确实验闭环。

4. 存在的问题：目前虽然主体系统已经实现，但论文中还有少量图和文字未完全补齐，特别是系统架构图、状态演进示意图、部分实验图及章节中的 TODO 标记尚未清理，影响整体完成度。  
   解决思路：接下来需要按章节列出待补内容并逐项消除。具体包括：补系统总架构图，补状态复用与校准过程示意图，补预测误差图，补跨表迁移对比图，统一检查所有图表编号、引用与正文描述是否一致。这样可以避免到论文后期同时处理“实验补数”和“文稿修补”两条线，减少返工。

5. 存在的问题：当前实验已经证明 TTFT 优化明显，但还需要进一步梳理论文最终主结论，明确哪些结论是“已经被数据充分支持”的，哪些只是“观察到趋势但还需要补实验”。如果这一步不提前做，后期容易导致论文结论过宽或表述失衡。  
   解决思路：接下来在补实验的同时同步维护一份“结论-证据”对照表。例如，“预测驱动调度显著改善 TTFT”已有完整端到端数据支撑；“跨模型预测表可以迁移”需要等待跨表实验补齐；“预测模块大规模在线可用”需要等待扩展性实验补齐。这样做可以让后续摘要、总结和答辩陈述都建立在已验证事实之上。
